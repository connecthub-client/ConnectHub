# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

**ConnectHub** is a modern cross-platform desktop SSH client (Tauri 2 + Rust + React): SSH terminal, SFTP, per-host VPN, and workspace/host management, backed by a local encrypted vault. MIT licensed. Public docs live at the repo root (`README.md`, `ARCHITECTURE.md`, `BUILD.md`, `INSTALL.md`, `DEVELOPMENT.md`, `ROADMAP.md`, `CHANGELOG.md`, `CONTRIBUTING.md`) — this file is Claude-Code-specific guidance and duplicates some of that content deliberately, since the intended audience and level of implementation detail differ.

## Release workflow

Versioning follows [SemVer](https://semver.org). A release bumps the version in **three** places (they are not derived from one another and must be kept in sync manually): `package.json`, `src-tauri/Cargo.toml` (`[package].version`), and `src-tauri/tauri.conf.json` (`.version`). Then:

1. Add a new section to `CHANGELOG.md` (move anything under `[Unreleased]` into it), following the existing Added/Changed/Fixed/Security grouping.
2. Commit the version bump + changelog as its own commit.
3. Tag `vX.Y.Z` and push the tag.
4. Build platform bundles (see `BUILD.md`) and attach them to a GitHub Release, along with a checksums file.

There is no automated release pipeline yet (see `ROADMAP.md`); `.github/workflows/ci.yml` runs type-checks/tests/clippy on push/PR but does not build or publish release artifacts.

## Repository conventions

- Commit messages: one imperative-mood, capitalized summary line, no enforced prefix (no `feat:`/`fix:` scoping) — see existing `git log` for the established style, and `CONTRIBUTING.md`.
- Community health files (`CODE_OF_CONDUCT.md`, `CONTRIBUTING.md`, `SECURITY.md`, `SUPPORT.md`) and issue/PR templates (`.github/ISSUE_TEMPLATE/`, `.github/PULL_REQUEST_TEMPLATE.md`) already exist at their standard GitHub locations — don't recreate or relocate them.
- Report/handle security-sensitive findings per `SECURITY.md` (private GitHub advisory), never as a public issue.

```bash
npm install

```

```bash
npm run build               # frontend only: tsc && vite build

```

```bash
cargo check                             # fast compile check
cargo test --lib                        # unit tests (fast, in-memory SQLite, no network)
cargo test --lib -- --ignored           # live integration tests against a real local sshd (see below)
cargo test --lib <test_name_substring>  # run a single test or module
cargo clippy --lib --no-default-features
```

```bash
npx tsc --noEmit    # type-check only
```

There is no configured lint/format command for the frontend beyond `tsc`.

## Live integration tests

Tests under `*::live_sshd_tests` modules (in `ssh/session.rs`, `ssh/sftp.rs`, `ssh/exec.rs`) are marked `#[ignore]` and connect to a **real local `sshd` on port 22** using a dedicated throwaway SSH keypair whose public half is appended to `~/.ssh/authorized_keys`. They hardcode the private key path to a location under the Claude session's scratchpad directory — when regenerating this environment, recreate that keypair and authorized_keys entry before running `cargo test --lib -- --ignored`, and never reuse or touch the real `~/.ssh` keys already present on the machine.

## Architecture

Tauri 2 desktop app: Rust backend (`src-tauri/`) exposes `#[tauri::command]` functions; React frontend (`src/`) calls them exclusively through a typed bridge layer, never `invoke()` directly from components.

The product has been renamed twice (SSH Tool → Termora → **ConnectHub**, current). User-facing branding (`productName`/`identifier`/window title in `tauri.conf.json`, `index.html`, Cargo package/lib names, the Google OAuth success page, the polkit dialog's description/message text) tracks the current name. A handful of internal, non-user-facing identifiers deliberately still say "termora"/"sshtool" and are not renamed on principle, each with a comment at its definition explaining why: `vault/store.rs::db_path`'s `"sshtool"` data directory, `google/mod.rs`'s Drive backup filenames (`termora-vault.db`/`termora-local-secret` - renaming would orphan existing backups), and `vpn/setup.rs`'s `HELPER_PATH`/`POLICY_PATH`/`POLICY_ACTION_ID` (renaming would orphan an already-installed polkit rule and silently force re-running VPN setup). If the product is ever renamed again, leave these alone too.

### Backend layering (`src-tauri/src/`)
- `commands/*_commands.rs` — thin `#[tauri::command]` wrappers. They lock `state.db`/`state.vault_key` and delegate to `data/`; no business logic lives here.
- `data/*.rs` — CRUD and domain logic against SQLite (`rusqlite`), one module per entity (`hosts`, `groups`, `identities`, `ssh_keys`, `snippets`, `vpn_profiles`, `host_csv`). Each has its own `#[cfg(test)] mod tests` using an in-memory `Connection`.
- `models/*.rs` — plain `Host`/`Identity`/`SshKey`/etc. structs plus their `*Input` create/update counterparts (serde `Deserialize`, snake_case fields — Tauri does NOT camelCase these).
- `ssh/*.rs` — the actual SSH functionality via `russh`/`russh-sftp`: `session.rs` (interactive PTY sessions + the shared `connect_and_authenticate` helper reused by SFTP/exec/stats), `sftp.rs`, `exec.rs` (one-off command execution, used by snippets' run-on-hosts and `stats.rs`), `stats.rs` (per-host CPU/RAM/swap/disk/net sampling for the Performance panel), `known_hosts.rs` (TOFU host-key pinning).
- `vault/` — `kdf.rs` (Argon2id), `crypto.rs` (AES-256-GCM field-level encrypt/decrypt), `store.rs` (vault create/unlock/auto-unlock, the per-install local secret, and the one-time legacy-password migration; resolves the SQLite db path).
- `google/` — `oauth.rs` (PKCE authorization-code flow via a loopback `TcpListener`, token exchange/refresh), `drive.rs` (`appDataFolder` upload/download/find via Drive API v3), `mod.rs` (ties both together into `login`/`backup_now`/`restore_from_drive`, called by `commands/backup_commands.rs`).
- `vpn/` — `setup.rs` (installs a narrowly-scoped polkit rule + helper script so openvpn can be launched as root without a per-connect password prompt), `mod.rs` (spawns/tracks live openvpn processes per profile via `VpnMap`, an `Arc<DashMap<Uuid, ActiveVpn>>` keyed by profile id, controlled over openvpn's local TCP management interface rather than OS signals - see "VPN profiles" below).
- `state.rs` — `AppState`: holds the `Mutex<Connection>`, the in-memory `Mutex<Option<VaultKey>>`, and `DashMap`s of live SSH/SFTP sessions keyed by UUID.
- `error.rs` — single `AppError` enum (`thiserror`) shared by every command; serializes to a plain string for the frontend.

New commands must be registered in **two** places in `lib.rs`: the `use commands::...` import list and the `tauri::generate_handler![...]` macro call — easy to forget the second one.

`ssh::session::ClientHandler::check_server_key` (the TOFU host-key check) deliberately opens its own short-lived `rusqlite::Connection` to the vault db file rather than going through `AppState.db`'s `Mutex` - `ClientHandler` is a long-lived object for the whole SSH session, not a single command, so threading a shared `Mutex<Connection>` into it would entangle it with `AppState` for little benefit. The cost is that `known_hosts::verify_or_trust`'s SELECT-then-INSERT (keyed on `(hostname, port)`) is *not* automatically serialized against other connections the way every `data/*.rs` call is - two hosts being connected to for the first time at once (or the same never-before-seen host from two tabs, e.g. Connect then SFTP in quick succession) run this on two fully independent connections. It's made safe with SQLite itself rather than an app-level lock: `verify_or_trust` wraps the check-then-insert in a `BEGIN IMMEDIATE` transaction (acquires the write lock before the SELECT, so a second concurrent call can't interleave between the SELECT and the INSERT), and both this ad hoc connection and the main `AppState.db` connection (`vault::store::open_at`) set a `busy_timeout` - without it, `BEGIN IMMEDIATE` or any other write would fail immediately with `SQLITE_BUSY` on the slightest overlap instead of just waiting the few milliseconds needed.

### Data model
`Group` (nested via `parent_id`) → `Host` (references `Identity`, optional `vpn_profile_id` pointing at a `VpnProfile`) → `Identity` (username + auth method, references `SshKey`) → `SshKey`. Plus standalone `Snippet` and `VpnProfile`, and a single-row `google_auth` table (see below). `hosts.vpn_profile_id` was added after the table already existed in shipped databases, so `data::init_schema` backfills it by hand (`add_column_if_missing`) rather than relying on `CREATE TABLE IF NOT EXISTS`, which is a no-op against an existing table - the pattern to follow if a future column needs adding to an existing table again.

Only secrets are encrypted at the field level (identity passwords, private keys, key passphrases) via AES-256-GCM; everything else (labels, hostnames, ports, notes) is plaintext in SQLite for fast querying. `host_csv.rs` exports/imports this data model as CSV — it deliberately excludes all secret material, matching identities on the importing side by username/label rather than re-creating credentials.

### Vault / "master password"
There is **no user-facing master password**. `App.tsx` calls `vault_auto_unlock` on launch, which (`vault/store.rs::auto_unlock`) unlocks using a random secret generated once per installation and stored at `~/.local/share/sshtool/.local_secret` (`0600` permissions, never committed to source). If a vault was created before this scheme existed, `auto_unlock` transparently falls back to the old hardcoded password, then runs `migrate_to_new_secret` once to re-encrypt every identity/key secret under the new per-install key and retire the old password — this migration path must not be removed while any user could still be on the old scheme. The Argon2id/AES-256-GCM machinery in `vault/` is otherwise unchanged. The SQLite file lives at `dirs::data_dir()/sshtool/vault.db` (hardcoded literal `"sshtool"` in `vault/store.rs::db_path`, independent of the Tauri app identifier/product name — renaming the app in `tauri.conf.json` does not move or affect this path).

### VPN profiles
`Host.vpn_profile_id` is a nullable FK to `vpn_profiles` (many hosts can share one profile - e.g. one office VPN unlocking a whole private subnet). Connecting requires root (a new tun interface + routes), which the app never has directly, so `vpn::connect` shells out via `pkexec` to a helper script installed once by `vpn::setup::install()` - see the VPN setup section in the README for the full rationale. The helper re-validates its own arguments at runtime (config/auth files must live under the calling user's own `~/.local/share/sshtool/vpn-profiles/`, looked up via `$PKEXEC_UID` since pkexec resets `$HOME`) and unconditionally forces `--script-security 0`, so an uploaded `.ovpn`'s `up`/`down`/`route-up` directives can never execute code as root regardless of what's in the file - that flag, not the polkit rule itself, is the actual security boundary. Once running, `vpn::mod.rs` controls the (root-owned) openvpn process over its local TCP **management interface** rather than OS signals, since an unprivileged process can't `kill()` a root-owned one: connecting opens a `127.0.0.1` socket openvpn was told to bind, sends `state on` to get async `>STATE:...` notifications, and disconnecting sends `signal SIGTERM` over that same socket. `vpn::connect` blocks (with a timeout) until the first CONNECTED/error transition so the frontend gets a definitive result instead of polling, while a background task keeps monitoring the socket afterward so a later status query reflects an unexpected drop.

`VpnProfile.avoid_default_route` (column default `1`, backfilled via `add_column_if_missing` the same way as `hosts.vpn_profile_id`) filters a pushed `redirect-gateway` back out via `vpn::effective_config` appending `pull-filter ignore "redirect-gateway"` to the config actually written to disk. This was the first attempt at letting multiple profiles run at once, added after real multi-VPN testing showed two profiles fighting over who owns `0.0.0.0/0` (second one failing to connect, or the whole machine's internet breaking). It turned out insufficient on its own: real profiles from the same testing relied on the server's own `redirect-gateway`/routing behavior to reach their target at all (one had `hostname` = a public IP, meaning the VPN's actual job was changing the exit IP a firewall trusts, not just reaching a private subnet) - forcing split-tunnel broke those specific hosts rather than fixing the conflict. The checkbox still exists (default on for new profiles) but now only affects the profile's *other*, unrelated traffic - it is no longer what makes an assigned host reachable.

What actually makes concurrent multi-VPN work is `vpn::add_host_routes`, called from `run_vpn` the moment a `>STATE:...,CONNECTED,...` line is seen: it looks up every host referencing this profile (`data::hosts::list_by_vpn_profile`, snapshotted by `vpn::connect` *before* openvpn even starts), resolves each hostname to every IPv4 address it has (`resolve_all_ipv4` - literal IPs short-circuit, real hostnames go through a bounded `tokio::net::lookup_host`), finds which `tun*` interface was just assigned the local tunnel IP from that STATE line (`extract_local_tunnel_ip` reads the 4th CSV field; `find_tun_interface` shells out to the unprivileged `ip -4 -o addr show` to match it to an interface name), and adds a `/32` route for each resolved IP through that interface (the actual per-hostname work is factored into `add_routes_for_hostname`, shared with `ensure_host_route` below). A `/32` route always outranks a broader one in the kernel's routing decision, so this works regardless of what either VPN server pushes for routing and regardless of which profile (if either) currently holds the default route - the actual bug the `avoid_default_route` pull-filter was trying and failing to solve. Everything here is best-effort (failures don't affect the VPN's own reported status): worst case, reachability falls back to whatever the server/OS would have done anyway.

Because `add_host_routes` only runs once, using whichever hosts referenced the profile at that exact moment, a host added to (or assigned) the profile *after* its VPN already connected would otherwise never get a route until the VPN was manually disconnected and reconnected - a real bug hit in practice with more than one host sharing a profile (see `data::hosts::tests::list_by_vpn_profile_returns_only_hosts_sharing_that_profile`, which explicitly covers 3 hosts on the same profile, not just 2 - "handle any number of hosts sharing a profile" is a real requirement here, not just the pair originally reported). `vpn::ensure_host_route(state, vpn_map, host_id)` closes that gap: `ActiveVpn` now also holds `tun_iface: Arc<Mutex<Option<String>>>` (populated once by `add_host_routes` the moment the interface is known) and `added_routes` itself (previously only a value captured in `run_vpn`'s closure, invisible elsewhere), so a route can be added on demand for a single host without needing to rediscover the interface.

There's a second, related race worth knowing if this area changes again: `vpn::connect()`'s early-return guard (`if matches!(current.state, Connecting | Connected) { return Ok(current); }`) means a caller whose `ensureVpnUp` runs while a *different* host's connect attempt for the same profile is still `Connecting` gets back that in-flight snapshot immediately, not the eventual outcome - `vpnStore.ts::ensureVpnUp` handles this by polling (re-calling `connect`, cheap/side-effect-free per that same guard while still `Connecting`) up to a bounded timeout rather than treating "still connecting" as a hard failure, and calls `vpnEnsureHostRoute` on **every** path that ends with the profile confirmed connected (both the already-connected short-circuit and the branch that just connected it), not only the short-circuit - a host whose own connect attempt merely piggy-backed on someone else's in-flight one still needs its route added once that other attempt actually finishes.

A third race lived one level below that guard: the guard's `vpn_map.get(&profile_id)` check and the eventual `vpn_map.insert(...)` (after db reads, config/auth file writes, and spawning the pkexec helper) used to be two separate DashMap operations with real async work in between - so several hosts sharing one *not-yet-connected* profile, gated concurrently (Snippets' "run on hosts" is the easiest way to trigger this, since it gates every selected host in parallel), could each observe the profile as absent and each launch its own openvpn process, silently clobbering each other's `ActiveVpn` entry and leaking every process but the last. `vpn::try_claim` closes this by making the check-and-insert one atomic DashMap `entry()` match: only whichever caller observes `Entry::Vacant` proceeds to actually connect (constructing the `ActiveVpn` synchronously and inserting it before any of the fallible setup runs), and it's now `Entry::Occupied` for `Connecting`, `Connected`, *and* `Disconnecting` alike (previously only the first two short-circuited, so calling `connect()` mid-disconnect could relaunch a duplicate process too). Since the claim itself happens before the fallible setup (`start_connection`), a failure there (`?`-propagated) explicitly `vpn_map.remove(&profile_id)`s the speculative claim - otherwise a failed connect attempt would leave a phantom "Connecting" entry in the map forever, since `run_vpn`'s own `cleanup()` is never reached for a connection that never actually launched.

This needs a second privileged helper (`vpn::setup::ROUTE_HELPER_PATH` / `ROUTE_POLICY_PATH` / `ROUTE_POLICY_ACTION_ID`) alongside the openvpn one, since adding a route needs root too - narrowly scoped to `ip route replace <ip>/32 dev <tun*>` with both arguments validated in the shell script itself. Unlike the openvpn helper/policy (kept as literal `termora-*` strings - see the rename note above), these are genuinely new, so they use the current `connecthub-*`/`com.connecthub.*` naming from the start. `setup::is_installed()` now checks for all four files, so anyone who ran setup before this existed sees the one-time setup banner again exactly once (to add the missing route helper) without disturbing the already-working openvpn helper/policy or any already-connected profile.

VPN is entirely automatic from the user's perspective - there's no manual VPN button anywhere. The gating logic lives in **one place**, `vpnStore.ts::ensureVpnUp(host): Promise<{ ok: boolean; message?: string }>`: if the host has a `vpn_profile_id`, it connects the VPN if needed (polling through the in-flight race described above) and makes sure this specific host has its route (`vpnEnsureHostRoute`) before resolving `ok: true` - every caller that's about to reach a host over the network goes through this rather than each needing its own copy of the check. Callers: `AppShell.tsx::ensureVpnUp` wraps it to additionally set the Connect/SFTP inline busy/error UI state (`vpnGateHostId`/`vpnGateError`, scoped to the currently-displayed host by id so an error connecting one host can't linger against a different one you navigate to afterward) - `handleConnect`/`handleOpenSftp` call that wrapper before opening a session, and `HostTree.tsx`'s double-click/right-click "Connect" reach it the same way (both call `onConnectHost` → `AppShell`'s handler directly, never a bypass). `RunSnippetForm.tsx` (Snippets "Run on hosts") calls the shared action directly for every selected host in parallel - a host whose gate fails is skipped (synthesized as its own `HostExecResult` error) while `runOnHosts` still runs against the rest, rather than aborting the whole batch over one unreachable host. `HostContextPanel.tsx::handleQuickCommand`'s one-off-exec fallback (only reached when no terminal session is already open for that host - the live-session branch is gated transitively, since a session couldn't exist without already having passed the gate) also calls it directly, with its own local error state independent of the `vpnBusy`/`vpnError` props (which remain scoped to the Connect/SFTP buttons specifically, to avoid cross-talk between unrelated UI). This design (one shared store action, thin per-caller wrappers for UI-specific error display) is deliberate - an earlier version had the check only inside `HostContextPanel`'s button handlers, which missed `HostTree.tsx`'s paths entirely, and separately, Snippets/Quick-Commands' exec path bypassed gating altogether until this was found and fixed - both were symptoms of the same mistake (gating logic duplicated or skipped per-caller) this shared-action structure is meant to prevent recurring. On the way back down, `vpnStore.ts::releaseIfUnused(hostId)` is called whenever a session tab closes (`AppShell.tsx::handleCloseTab`) - it disconnects that host's VPN profile only if no *other* open session still references it, so one shared profile stays up as long as anything needs it and comes down automatically once nothing does. `vpn::disconnect_all` (wired to Tauri's `RunEvent::ExitRequested` in `lib.rs`, plus a manual "Disconnect all" button in `VpnPanel.tsx`) is the safety net for anything left connected when the app closes or gets stuck - a forgotten/orphaned VPN connection silently rerouting traffic (including breaking the user's own internet, if the profile pushes a default-route redirect) was an actual bug hit in testing before this existed. Editing a host (`hostsStore.ts::updateHost`) also best-effort calls `vpnEnsureHostRoute` if the saved host has a `vpn_profile_id`, so a changed `hostname`/`vpn_profile_id` gets its route refreshed immediately rather than waiting for the next full VPN reconnect.

### Google Drive backup
Settings → Backup lets a user sign in with their own Google account (OAuth2 PKCE, loopback redirect, `google/oauth.rs`) and back up/restore the **entire vault file** plus the local secret to a hidden Drive `appDataFolder` (`google/drive.rs`), invisible in the user's normal Drive UI and readable only by this app. `google/mod.rs::backup_now` snapshots the live SQLite connection via `rusqlite::backup::Backup` (not a raw file read, since another connection has the file open) before uploading; `restore_from_drive` downloads both files, temporarily swaps `AppState.db` to an in-memory connection to release the file handle, overwrites the real db file and local secret, reopens, and re-runs `auto_unlock`. The refresh token is stored *inside* the vault (`google_auth` table) so it survives a restore. `google::oauth::CLIENT_ID`/`CLIENT_SECRET` are real, working credentials for this project's own registered "Desktop app" OAuth client (Drive API enabled) - safe to keep public because the actual security boundary is PKCE (a fresh `code_verifier` per sign-in, never persisted or transmitted except as a one-way hash), not secrecy of these values; see the comment at their definition and the README's "Google backup setup" section for the full reasoning and how to swap in a different client if forking.

Sign-in is cancellable: `oauth.rs::login`/`await_redirect` use an async `tokio::net::TcpListener` + `tokio::select!` racing `listener.accept()` against a `oneshot::Receiver<()>` cancel signal and a `LOGIN_TIMEOUT` (180s) safety-net sleep. `AppState.google_login_cancel: Mutex<Option<oneshot::Sender<()>>>` holds the sender while a sign-in is pending; `google::cancel_login(state)` (Tauri command `google_login_cancel`) fires it if present, a no-op otherwise. This exists because closing the browser tab mid-flow gives the app no other signal that sign-in was abandoned — the frontend's "Cancel" button next to the "Waiting for sign-in…" state (`GoogleBackupSection.tsx`) is the only way to get unstuck short of the timeout.

### Frontend (`src/`)
- `lib/tauri-bridge/` — one file per domain, each just wrapping `invoke("command_name", { args })`; `types.ts` holds the shared TS interfaces mirroring the Rust models. Always add new bridge functions here rather than calling `invoke` from components.
- `state/*Store.ts` — zustand stores. Mutations generally `await` the backend call then re-fetch the full collection (`loadAll()`) rather than patching state in place, since collections are small and this sidesteps subtle bugs from `ON DELETE SET NULL` cascades. `settingsStore.ts`'s theme defaults to `"dark"`.
- `pages/AppShell.tsx` — the main layout/router-equivalent: owns `mainView` (which manage-tab or which open session tab is active) and `modal` (which create/edit form is open) as local state; every panel/form is a controlled child. It also derives `contextHost` (the host shown in the right-side panel) by looking it up **live** from the `hosts` array by id — never hold onto a session's captured `host` snapshot directly for display, since fields like `last_connected_at` change after the snapshot was taken.
- `components/panels/HostContextPanel.tsx` — persistent right-side panel for the selected/active host, each section its own rounded card: Connect (disabled once already connected)/SFTP actions, a **Host Details** card (address/port/user/VPN profile/live session status/key/last connected - independently show/hide toggleable, `settingsStore.hostDetailsVisible`), a toggleable **Performance** card (live CPU/RAM/swap/disk/net once connected, polled via `ssh::stats`, `settingsStore.performancePanelVisible`), and a **Quick Commands** card. Quick Commands has its own **Auto-Run** switch (`settingsStore.quickCommandAutoRun`): ON writes a clicked command straight into the host's live terminal session followed by Enter (via `sessionsStore.sessionIds` - the backend session id `TerminalView.tsx` publishes once connected, keyed by tab id, so this panel can `sessionWrite` into a session it doesn't own); OFF writes the text only, leaving it in the terminal's input line for the user to edit/submit themselves. With no terminal session open for that host, it falls back to the one-off exec channel instead (inline result shown in the panel). Above the saved-snippet list sits a **Most used** ranking: prefers `commandHistoryStore.ts`'s `remoteTopUsed` (the target server's own `~/.zsh_history`/`~/.bash_history`, fetched fresh by `TerminalView.tsx` on every connect and ranked by frequency - `parseRemoteHistory`/`rankRemoteHistory`), falling back to `topUsedCommands` over locally-recorded history (capped at 100 entries/host) when the remote fetch finds nothing (restricted account, no history file, non-bash/zsh shell). Local history itself comes from two sources: clicking a snippet/Most-used entry here, and `TerminalView.tsx` re-deriving "lines" from the user's raw keystrokes sent to the remote shell. The terminal-typed path is a best-effort heuristic, not real shell integration: it skips whole escape sequences (arrow/function keys, bracketed paste), treats Ctrl+C/Ctrl+U as discarding the in-progress line rather than submitting it, and - since this history is persisted locally in the browser's storage without the vault's encryption - tries to avoid recording likely secrets by suppressing the line typed immediately after remote output matching a password/passphrase prompt (`TerminalView.tsx`'s `SECRET_PROMPT_RE`, deliberately just the bare words rather than requiring a trailing colon, since real prompts vary too much and a false positive there is far cheaper than a leaked secret). This is not foolproof (no true way to distinguish a secret prompt from the client side), so treat it as a mitigation, not a guarantee, if extending this area. There's no "Edit host" shortcut or "Recent" run-history list here anymore - editing is via the host tree's right-click menu, and "Recent" was superseded by "Most used".
- `components/panels/GoogleBackupSection.tsx` — the Settings → Backup UI (sign in/out, back up now, restore) built on `lib/tauri-bridge/backup.ts`; a destructive restore requires an explicit `confirm()` and reloads the window afterward rather than trying to patch every zustand store in place.
- `state/vpnStore.ts` + `components/panels/VpnPanel.tsx` / `components/forms/VpnProfileForm.tsx` — VPN profile CRUD follows the `hostsStore.ts` refetch-after-mutation pattern, but connection status (`statuses: Record<profileId, VpnStatus>`) is tracked and refreshed separately via `refreshActive()`/`vpn_active_statuses`, since it changes independently (and asynchronously, mid-connect) from the profile records themselves. Both `VpnPanel` and `HostContextPanel` poll `refreshActive` on a short interval only while at least one profile is `connecting`/`disconnecting`, not continuously. `vpnStore.ts` also imports `hostsStore`/`sessionsStore` (one-directional only - nothing imports back) to answer "is any other open session still using this profile" for `releaseIfUnused`. `HostForm.tsx`'s VPN section is a three-way None/Upload/Use-saved toggle (mirroring its existing identity new/existing toggle) - "Upload" creates the `VpnProfile` inline via `vpnStore.createProfile` at submit time rather than requiring a separate trip to the VPN tab first. `HostForm.tsx`'s private-key auth section has an analogous two-way "Use saved key" / "Import new key" toggle (only shown once at least one key exists; import fields show directly otherwise) - "Import new key" calls `hostsStore.importKey` at submit time the same way the VPN toggle creates a profile inline.
- `pages/AppShell.tsx` session tabs support drag-to-reorder via native HTML5 DnD (`draggable`, `onDragStart`/`onDragOver`/`onDrop`) - `sessionsStore.ts::reorderSessions(fromIndex, toIndex)` splices `openSessions`. No external DnD library. Note for testing: synthetic mouse events (e.g. `xdotool` under X11) can trigger `dragstart` but generally cannot complete a `drop` in WebKitGTK - this is an automation-tooling limitation, not a signal that the feature is broken; verify manually if GUI-automating this area.
- `components/common/Modal.tsx` — the backdrop itself scrolls (`overflow-y-auto` on the fixed wrapper, `mx-auto my-8` instead of `flex items-center` on the dialog box) rather than the dialog box being vertically centered with no scroll path - centering-without-scroll was found to make a sufficiently tall form (`HostForm` once the VPN profile field was added) unreachable from its top on an 800x600 window, with no way to scroll up to it. Keep this in mind for any future modal content that could grow tall.
- `components/terminal/TerminalView.tsx` + `components/sftp/SftpBrowser.tsx` — one instance per open session tab, kept mounted (via CSS `visibility`, not `display:none`/unmount) while switching tabs so the SSH connection and xterm.js scrollback survive. `display:none` was tried first and silently drops the most recent scrollback line in WebKitGTK — don't reintroduce it.
- `components/sidebar/HostTree.tsx` — double-click a host row to connect (only if it has an identity); right-click opens a small custom context menu (Connect/Duplicate/Edit/Delete) built with a `useEffect` closing it on any outside click/contextmenu/keydown; a status dot per row reflects whether a session is currently open for that host. Neither of these connect paths (nor the Hosts grid's double-click) checks for an already-open session itself - that guard lives in `AppShell.tsx::handleConnect` (the function both eventually call), which switches to the existing terminal tab instead of opening a duplicate one. Keep it there rather than re-adding a per-caller check: only `HostContextPanel`'s own Connect button used to guard this (via its `disabled`/`sessionOpen` prop), so double-clicking or right-clicking a host from the tree (or grid) that already had a session open used to silently open a second, fully redundant one. A search box filters the tree by label/hostname; above the group tree, collapsible **Favorites** (`host_set_favorite`/`hosts.is_favorite`, toggled via a star on hover) and **Recent** (the 5 most recently connected, by `last_connected_at`) sections list matching hosts flat, depth 0. `components/common/hostIcons.tsx` supplies the preset icon set (generic/cloud-provider/A-Z monogram) rendered next to each host's label wherever `host.icon`/`host.color` are shown (tree rows, the context panel header, the center Hosts grid).
- Native file dialogs use `@tauri-apps/plugin-dialog` (`open`/`save`); reading/writing the chosen path goes through `local_read_text_file`/`local_write_text_file` Tauri commands rather than the `fs` plugin, matching the existing `local_fs_commands.rs` pattern.
- `components/layout/ActivityBar.tsx` — the VSCode-style icon rail: a chevron pinned at its own top (not inside the Primary Side Bar) toggles `settingsStore.leftSidebarVisible` and is always rendered regardless of which destination is active, so it's reachable even when the sidebar itself isn't mounted; below it, clicking the already-active destination icon also toggles the sidebar (`AppShell.tsx::handleActivitySelect`), clicking a different one switches and force-shows it.
- `pages/AppShell.tsx`'s center "Hosts" view (when no session tab is focused) renders every host as a grid of cards - click selects (mirrors the sidebar), double-click connects (same identity-required guard as `HostTree.tsx`) - so hosts stay reachable when the Primary Side Bar is collapsed.
- Color palette is teal-accented and dark-first, built on Tailwind's `slate` scale (not `neutral`, and not a blue accent) — reuse `teal-*`/`slate-*` for new interactive elements rather than introducing another palette. Body font is self-hosted Inter (`@fontsource/inter`, imported in `main.tsx`); give checkboxes/radios/range inputs an explicit `accent-teal-600` rather than leaving them at the browser/OS default accent color.

### Known WebKitGTK quirks (Linux)
- Native `<select>` elements render using the OS GTK theme rather than the page's CSS and can end up with unreadable (e.g. white-on-white) text when the app's theme diverges from the system theme. Fixed via the `.themed-select` CSS class (`App.css`) + `appearance: none` with a hand-drawn arrow — reuse this class for any new `<select>`, don't rely on native styling.
- `color-scheme` must be set explicitly (`App.tsx`'s theme effect) for native form controls to follow the app's manual light/dark override instead of the OS preference.
